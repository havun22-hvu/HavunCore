# ADR 007 — MCP-laag + Multi-Claude Orchestration verwijderd

## Status

**Geaccepteerd 2026-05-09.** Vervolg op [ADR 004](004-vision-orchestration.md) (DEPRECATED 2026-03-01) — codebase is nu in lijn met die deprecated-status.

## Context

ADR 004 beschreef een Multi-Claude Orchestration vision waarin HavunCore via een externe Node-bridge (`havun-mcp` op `localhost:3000`) cross-project messages publiceerde. Andere project-Claudes lazen die berichten en pickten taken op. ADR 004 is op 2026-03-01 als DEPRECATED gemarkeerd, maar de codebase werd nooit opgeruimd.

Inspectie op 2026-05-09 (Aeterna-sessie + cross-check):

- **Externe Node-bridge** is al maanden uit (connection-refused op `localhost:3000`).
- **Geen ander productie-project** roept `/api/mcp/messages` aan (grep over alle `/var/www/*` checkouts).
- **Geen crontab of scheduler** triggert `havun:orchestrate` of `havun:tasks:*`.
- **Geen externe composer-consumer** gebruikt `Havun\Core\Services\MCPService`, `TaskOrchestrator`, `MCPMessage` of `APIContractRegistry`.
- **Productie-tabel `mcp_messages`** bevat 10 historische rows zonder recente readers.
- **PHP-callsites** vingen connection-refused met try/catch + log-warning — features waren silent-degraded, niet "werkend zonder MCP".

Conclusie: zowel de transport-laag (MCP) als de feature die erop steunde (Multi-Claude Orchestration) zijn dood — geen consumers, geen lezers, geen schedulers.

## Beslissing

**Verwijder de hele laag.** Geen vervanging, geen Queue-fundament op voorraad, geen rename naar `cross_project_events` — de use-case bestaat niet meer.

## Waarom geen vervanging

Tijdens de Aeterna-sessie was Laravel Queue voorgesteld als drop-in vervanging voor MCP. Niet doorgevoerd, want:

- **HavunCore heeft geen queue-fundament** — `QUEUE_CONNECTION=sync` op productie, geen workers, geen `config/queue.php`. Queue introduceren = nieuwe ops-overhead (driver-keuze, systemd-unit, failed-jobs-monitoring) zonder concrete use-case.
- **Queue ≠ pub/sub** — MCP was multi-consumer broadcast, Laravel Queue is producer→single-worker. Conceptueel mismatch.
- **YAGNI** — wacht met queue-fundament tot een echte use-case zich meldt; ontwerp dan voor die use-case ipv MCP-symmetrie.

Voor toekomstige cross-project communicatie:
- **Passieve kennis** (status, patterns, beslissingen) → KB via `docs:search` (al actief)
- **Actieve coördinatie** (als en wanneer nodig) → Laravel Queue met database-driver, dan bewust ontworpen
- **Eén Claude over de hele monorepo** vervangt grotendeels de cross-project messaging use-case

## Composer-package impact

Publieke API verwijderd: `Havun\Core\Services\MCPService`, `Havun\Core\Services\TaskOrchestrator`, `Havun\Core\Services\APIContractRegistry`, `Havun\Core\Models\MCPMessage`.

Inventaris bevestigde **0 externe consumers** in `havun22-hvu` portfolio. Composer-package krijgt **major-version-bump** (semver) bij volgende release om breaking-change discipline te respecteren.
