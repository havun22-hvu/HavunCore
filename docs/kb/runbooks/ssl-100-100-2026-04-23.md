# SSL Labs 100/100 — Key Exchange + Cipher Strength

> **Aanleiding:** SSL Labs herdenkingsportaal.nl: Key Exchange 90/100,
> Cipher Strength 90/100. Per `havun-quality-standards.md` is enterprise
> de norm — geen uitzonderingen op 90 accepteren als 100 zonder
> nadelige impact haalbaar is.

## Scope

Server: 188.245.159.115. Raakt alle 7 productie-domeinen — fix gebeurt
op nginx-cipher-config-niveau (cross-vhost) en cert-niveau (per domein,
maar via certbot 1 commando per domein).

## Plan

### Fase 1 — Voorbereiding

1. Backup huidige nginx-config + cert-paden (al gedaan in commit b950e0c
   gisterennacht — `/root/nginx-backup-2026-04-23.tar.gz`)
2. Verifieer certbot-versie en ECDSA-support (certbot 1.21+ heeft
   `--key-type ecdsa` flag native)

### Fase 2 — Cert herconfiguratie naar ECDSA

Per domein (7 stuks) opnieuw uitgeven met ECDSA-key:
```bash
certbot certonly --nginx \
    --key-type ecdsa --elliptic-curve secp384r1 \
    --cert-name herdenkingsportaal.nl \
    -d herdenkingsportaal.nl -d www.herdenkingsportaal.nl \
    --force-renewal
```

ECDSA secp384r1 = SSL Labs Key Exchange 100. Werkt op alle browsers
sinds ~2013. Smaller key (~50% kleiner dan RSA 4096), sneller handshake,
zelfde security-level.

### Fase 3 — Nginx ssl_ciphers hardening

Update `/etc/nginx/snippets/ssl-params.conf` (of vergelijkbaar):
```nginx
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;
ssl_prefer_server_ciphers on;
ssl_ecdh_curve secp384r1;
```

Alleen 256-bit AEAD ciphers (GCM/ChaCha20). Geen 128-bit, geen CBC,
geen RSA key-exchange. Cipher Strength → 100.

### Fase 4 — Validatie

1. `nginx -t` + `systemctl reload nginx`
2. Smoke-test 7 domeinen via curl (zelfde als gisteren)
3. SSL Labs handmatig opnieuw met "Clear cache" → verwacht **A+** met
   100/100 op Key Exchange en Cipher Strength

## Risico's

| Risico | Kans | Impact | Mitigatie |
|--------|------|--------|-----------|
| ECDSA-cert breekt oudere client | Zeer laag | Mid | Alle browsers sinds 2013 ondersteunen ECDSA |
| 256-bit-only sluit oudere TLS 1.2 clients buiten | Zeer laag | Mid | Doelgroep gebruikt geen pre-2014 devices |
| Certbot-renewal cron stopt werken | Laag | Hoog | Renewal config wordt automatisch aangepast |
| Nginx reload faalt | Zeer laag | Hoog | nginx -t test eerst |

## Uitvoer-resultaat (23-04-2026, ~06:25)

**Fase 2 — ECDSA certs ✅**
Alle 7 productie-domeinen herrouteerd naar ECDSA secp384r1 via certbot.
Certbot 5.5.0. Eerste poging faalde met `--reuse-key` blokkade —
opgelost door `--no-reuse-key` toe te voegen (key-type-wisseling
vereist nieuwe key).

**Fase 3 — Hardened SSL snippet ✅**
- `/etc/nginx/snippets/ssl-hardened.conf` aangemaakt:
  ```nginx
  ssl_protocols TLSv1.2 TLSv1.3;
  ssl_prefer_server_ciphers on;
  ssl_ciphers ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;
  ssl_ecdh_curve secp384r1;
  ssl_session_cache shared:SSL:10m;
  ssl_session_timeout 1d;
  ssl_session_tickets off;
  ```
- 7 vhosts geupdatet: `include /etc/letsencrypt/options-ssl-nginx.conf`
  vervangen door `include /etc/nginx/snippets/ssl-hardened.conf`
  (havuncore had eigen ssl_protocols regel — die vervangen door include)
- Tussentijdse duplicate-include hiccups opgelost via sed-cleanup
